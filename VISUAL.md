## 1. 커버 제작 — 세 경로 + 의사결정

### 1.1 경로 비교

| 경로 | 방식 | 비용 | 이 머신 | AI슬롭 리스크 | 권장 |
|---|---|---|---|---|---|
| **A. 코드 제너러티브** | `cover_render.py` (PIL/NumPy 절차적) | 무료 | ✅ 즉시 | 없음 | 기본 시그니처 |
| **B. AI 이미지** | 확산모델(로컬/API) 전체 생성 | 로컬=GPU / API=건당 | ❌ 로컬 불가 / API만 | 높음 | 옵션 |
| **C. 하이브리드** | AI·사진 *배경* + 코드 타이포·Reina·디더 레이어 | 배경원 따라 | ✅(배경만 외부) | 낮음 | **상향 권장** |

### 1.2 이 머신은 로컬 AI 이미지 불가

환경(CONTEXT §3): **전용 GPU 없음(UHD 610급 iGPU), torch CPU 전용.** 2026 로컬 확산모델 최소 VRAM:

| 모델(2026) | 최소 VRAM | 강점 | 라이선스 |
|---|---|---|---|
| SD 1.5 | 4 GB | 최경량·LoRA | 버전별 |
| SDXL | 6–8 GB | LoRA·ControlNet 최다 | 상업가능(확인) |
| Z-Image Turbo | 13–16 GB | 초고속 8스텝 | Apache 2.0 |
| FLUX.2 [klein] | ~13 GB | 빠름·신규 | BFL 상업 |
| FLUX.1/2 [dev]·Qwen-Image | ~24 GB | 최상·텍스트렌더 | 별도 |

→ **VRAM 0이라 SD1.5조차 비현실적. 로컬 AI 이미지 도입 불가 확정.** 필요 시 API 또는 별도 GPU 머신.

### 1.3 AI 이미지 — API 옵션(배경용, 선택)

| 용도 | 모델 | 비고 |
|---|---|---|
| 텍스트 렌더 | Ideogram·Imagen·FLUX·Qwen·GLM-Image | 이중언어 글자 강함 |
| 최상 품질 | GPT Image 2·FLUX.2 [max]·Nano Banana Pro | 건당 과금 |
| 다용도 | Seedream 5.0·FLUX.1.1 Pro | — |
| 빠른 반복 | FLUX schnell·Z-Image Turbo·Recraft | 1–8 스텝 |

대략 건당 $0.01~0.10, 키 필요. **우리는 제목·Reina를 코드로 박으므로 AI엔 글자 없는 배경만 맡긴다**(텍스트 렌더 모델조차 불필요).

### 1.4 권장 — 경로 C(하이브리드)

배경(제너러티브/AI/사진) 위에 항상 코드 레이어 합성: (필요시 글로우)→모티프→**Reina 사인**→제목→**디더**. AI의 풍부함 + 코드의 정확함 + 시그니처 일관성 = AI 슬롭 방어.

> **불변 룰:** 배경원이 무엇이든 **Reina 사인·디더·밝은 톤·64×64 생존**(§3)은 코드 레이어로 항상 보장.

## 3. 고정 디자인 표준 (모든 곡 공통 시그니처 — 불변)

2026 커버 서치: 단순 중앙 그라데이션+산세리프는 "옛날(2018)" 신호. 좋은 커버는 **64×64 타일 식별·캡션 없는 장르 신호·릴리스 비주얼 세계 공유**.

| 항목 | 표준 | 근거 |
|---|---|---|
| **밝은 톤** | 항상 밝은 베이스. **우울 톤 금지** | 선생님 룰 |
| **Reina 사인** | 우상단, Great Vibes 흘림, 흰 글자 + 딥 외곽선(번짐 글로우 금지) | 공통 시그니처 |
| **디더** | 삼각 ~1.5 LSB(소스 디더) | 밴딩=소스 문제 |
| **해상도** | 2560×1440 + 정사각 크롭(스트리밍) | §4.3·64² 생존 |
| **타이포** | 제목(KO/EN) 또렷·진하게, 본문 텍스트 최소 | 미니멀 |

이 다섯은 **어떤 스타일·배경원에서도 코드 레이어로 항상 보장.** 곡마다 *바뀌는 것*은 §4.

### 3.1 AI 배경 프롬프트 3층 원칙 (fal_bg v3)

`hades/fal_bg.py` 배경 프롬프트는 **SCENE / CAMERA / GRADE** 3층으로 고정한다.

| 층 | 역할 | 가변성 |
|---|---|---|
| **SCENE**(최전방) | `--scene` = **가사 유래 장면 앵커**. 앵커 근거 가사 줄번호를 `--anchor` 로 genlog 기록 | 곡별 |
| **CAMERA** | 구도·매체 — wide shot·airy·제목 여백·cinematic 16:9 | 고정 |
| **GRADE**(후방) | 색·노출 — `day`(하이키·쿨 블루) / `bright_nocturne`(밝은 야경, 딥블랙 금지) | 하우스 불변(2택) |

**규칙:**
- **파랑은 GRADE에서 나온다.** 하늘·물·바다 같은 환경 오브제는 **가사에 실재할 때만** SCENE에 넣는다(장면을 파랗게 만들려 바다/하늘을 억지 삽입 금지).
- **네거티브 말미 필수**(원표준 복원 2026-07-21). flux-2-pro 는 negative 필드가 없어 **프롬프트 말미에 고정 네거티브 문자열**로 명시한다: `No yellow tint, no sepia, no film wash, no haze, no muted grey, no gloom.` 긍정 GRADE와 병행. **조립 구조 = SCENE(가사 앵커) → CAMERA(구도) → 긍정 GRADE(`crisp high-key exposure, cool blue-leaning color grade, vivid clean saturation, clear transparent air`) → 네거티브 말미.** (A/B 실측 2변형×2시드 A 우세로 확정 — genlog 검증 구조와 동일)
- **야간 정서 곡**은 `--grade bright_nocturne`(딥블랙·침울 금지, 빛 살림).

### 3.2 무인 게이트1 (scene_check — 2026-07-03, 헌장 §1.4 사인오프)

**"게이트1 사람 승인 필수"는 자동 검증기 체계로 대체된다.** 커버 승인은 사람이 아니라 `hades/scene_check.py`(헤드리스 비전) + tone_check + 합성 무결성의 합의로 이뤄진다.

- **scene_check**: `claude -p --allowedTools Read`(모델 `claude-sonnet-5`)로 후보를 비전 검사 → JSON 3항목 **anchor_present · no_lyric_contradiction · no_face_no_text** 전부 true면 PASS. 입력=후보 이미지 + SCENE 앵커(가사 줄번호) + `lyrics_ko.txt`.
- **자동 승인**: `auto_gate1`(hades_state)이 후보들에 tone_check→scene_check→`cover_render --bg` 합성→무결성(2560×1440·JPEG) 검사 후 **첫 합격 후보를 `.cover_ok` 자동 서명**. cover_gate 훅은 유지(검증기 서명으로 통과).
- **예외(HOLD)**: scene_check 2연속 FAIL(또는 후보 전원 미합격) → `state=HOLD_COVER`, `_notify_hold`(폰 알림; 실제 푸시 인프라는 별도 TODO), 사람 개입 대기. 사람이 `.cover_ok` 승인 시 복귀.
- **재렌더 규칙**: 커버가 바뀌면(모션 빌드 등) 기존 `.cover_ok` 무효 → 재검증/재승인 필수(§7.4).
- **수동 유지**: 트랙 config `cover_gate_auto: false` 면 자동 승인 제외(예: radio, 모션 빌드 대기).
- **사후 표본검사**: 업로드 5건당 1건 `spot_check` 플래그를 중앙 원장(`upload_ledger.json`) + 매니페스트에 기록.

## 4. 커버 다양성 전략 (양산형 방지 — §2.5)

**문제:** 한 가지 룩만 반복하면 AI 슬롭처럼 보여 수익화 방어가 무너진다.
**해법:** *불변 시그니처(§3)* + *곡별 가변 4축* + *스타일 패밀리 선택*.

### 4.1 곡마다 바뀌는 4축
1. **스타일 패밀리** (아래 카탈로그에서 선택/혼합)
2. **팔레트** (밝은 톤 룰 안에서 곡 정서별로 — 여명/시티/파스텔/네온-소프트 등)
3. **모티프** (가사에서 추출한 오브제 — 곡마다 다른 `motif_<곡>` 함수)
4. **컴포지션** (포컬 위치·여백·라인아트 크기·시점)

### 4.2 스타일 패밀리 카탈로그
`cover_render.py`의 `style` 프리셋으로 구현. 각 패밀리는 그라데이션 룰·레이어 구성·라인아트 처리·팔레트군이 다르다.

| 스타일 | 시각 특성 | 적용 장르(예) | 구현 포인트 |
|---|---|---|---|
| **Luminous Dawn** | 여명 글로우·보케·빛의 라인아트, 공기감 | future soul, electro-R&B | 현재 기본(「아무말도」) |
| **Color-Field Geometric** | 큰 색면·기하 형태·강한 대비, 미니멀 | funk, 업템포 팝 | 바우하우스/스위스, 채도↑ |
| **Soft Grain Analog** | 따뜻한 그레인·페이퍼 텍스처·여백, 잔잔 | 발라드, 포크, 어쿠스틱 | 일본 포토북 감성, 크림 톤 |
| **Gradient Mesh Flow** | 흐르는 추상 메시·유동 곡선 | 앰비언트, 드림팝 | 몽환·유동, 형태 최소 |
| **Line Focal** | 단일 선화 오브제 + 광활한 여백 | R&B, 네오소울 | 모티프를 크게·단독으로 |
| **Bright Nocturne** | 밤이되 밝게: 네온 윤곽·소프트 라이트·도시 | 시티팝, 나이트드라이브 | 딥블랙 금지, 미드톤+글로우 |
| **Photographic / AI-bg** | 사진·AI 배경 + 코드 레이어(경로 C) | 장소가 구체적인 곡 | 시네마틱, 밝은 노출 |
| **Dreamscape 3D** | 스타일라이즈드 3D 렌더·환상 지형(떠 있는 지형·유리질 수면)·볼류메트릭 데이라이트 | 일렉트로닉, 드림팝, 컨셉 곡 | fal 프롬프트: "stylized 3D render, soft glossy materials, volumetric daylight" 계열. 톤 명령(밝음·블루·비비드) 유지 |
| **Soft Sculpt** | 부드러운 매트 재질 3D 일러스트·서브서피스 산란·얕은 심도 | 발라드, 잔잔한 곡 | "soft matte 3D illustration, subsurface scattering, gentle depth of field". 스튜디오/엔진명 사용 금지(기법 용어만) |
| **Anime Scene** | 애니메 배경화 스타일 씬(인물 없음 또는 비식별) | 인물/서사 중심 곡 | kkotboda 원칙 준용: 기법 용어만, anti-childlike, 성인 비율 |

### 4.3 다양성 워크플로우 (매 곡)
1. **canvas-design으로 새 미학 철학 1개 생성** — 매번 다른 무브먼트(기계적 반복 차단).
2. 그 철학 → **스타일 패밀리 선택/혼합** + 새 팔레트 + 새 모티프 + 컴포지션 결정(네비게이터).
3. `cover_render.py <곡> --style <패밀리>` 로 렌더 (style 프리셋이 레이어 구성을 바꿈).
4. **불변 시그니처(§3)는 자동 적용** — Reina·디더·밝은 톤·64² 생존.
5. 시안 검증 → 확정.

**로테이션 규칙 (패밀리 반복 방지):**
- **연속 2곡 같은 패밀리 금지** — 직전 발행 곡의 스타일 패밀리를 `upload_ledger.json` 기준으로 확인하고, 동일 패밀리를 연속 배정하지 않는다.
- **패밀리 배정은 가사 정서/서사 기준** — 장르 기본값으로 배정 금지(§4 lyrics-first 원칙 유지). "발라드니까 Soft Sculpt" 식의 기계적 매핑 금지, 곡의 정서·서사에서 도출.
- **불변 시그니처(§3: Reina 사인·디더·밝은 톤·64² 생존)는 전 계열 공통 적용** — 3D·애니메 계열을 포함해 어떤 패밀리에서도 코드 레이어로 항상 보장.

> **검증 질문:** "이전 곡 커버 옆에 놨을 때 *다른 작품*으로 보이는가? 동시에 *같은 아티스트(Reina)*로 읽히는가?" 둘 다 ✅여야 통과.

### 4.4 곡별 예시(차별화 확인)
- **「아무말도」** — Luminous Dawn / 여명 팔레트 / 벤치·그네 라인아트 / 좌하단 제목·여백 우측.
- **「관람차」(다음)** — *다른 패밀리 권장*: Bright Nocturne 또는 Line Focal / 밝은 시티 더스크 팔레트(시안·골드) / **관람차 원형 + 바다 수평선** 모티프 / 중앙 포컬. → 「아무말도」와 한눈에 다르되 Reina로 같은 작가.

### 4.5 `cover_render.py` 확장 방향
- `--style` 인자 + `STYLES` 레지스트리(패밀리별 그라데이션·레이어·라인처리 함수).
- `motif_<곡>` 함수는 곡별로 네비게이터가 추가.
- 배경 소스 가변: 제너러티브(기본) 또는 `--bg <경로>`(AI/사진, 경로 C).
- 시그니처 레이어(Reina·디더·타이포)는 style·bg와 무관하게 항상 마지막에 합성.

## 5. 자막 구현 표준 (정의 §3)

### 5.1 액션 자막 (검증 완료)
- libass 네이티브 + ffmpeg 한 패스. 줄 팝업(`\fad`+`\t`) + 단어 색채움(`\kf`).
- `\kf` 색: **Primary=하이라이트(노랑), Secondary=베이스(흰)** — 직관 반대(고정).
- `PlayResX/PlayResY` = 렌더 캔버스(2560×1440)와 일치.

### 5.2 정렬 — 가사 보유 → forced-align

| 방법 | 정밀도 | 설치 | CPU | 판단 |
|---|---|---|---|---|
| **torchaudio MMS_FA**(CTC) | 양호 | 가벼움 | ✅ | **실용 권장** — 1100+ 언어, 전사 불필요 |
| **MFA**(GMM-HMM) | **최상 ~10ms** | 무거움(Kaldi·음소사전) | ✅ | 정밀 천장(선택) |
| WhisperX | 중 | 중 | ✅ | 타임스탬프 부정확 多 → 비권장 |

- 권장: 가사 보유 → 전사·분리 생략, **MMS_FA 직접 CTC 정렬.** `.venv-align`(Python 3.11, torch CPU). 채팅 컨테이너 불가(로컬 전용).
- 천장: 보컬 간섭 심하면 demucs(htdemucs_ft) 분리 후 정렬, 또는 MFA 정밀화.

### 5.3 한·영 양버전
- KO 음원 정렬 → **EN 1:1 타임코드 상속**(같은 줄 수). `dual`: KO 위(노랑 카라오케) + EN 아래(크림), 색 분리.

### 5.4 폴백
- 균등분할 폴백 + `offset_ms`는 임시 검증용. **드리프트(뒤로 갈수록 벌어짐)는 offset으로 못 잡음** → 발행 마스터는 forced-align 필수.

## 7. BGM 모션 표준 v1 (인스트루멘털 트랙)

> 인스트루멘털(가사 없는 BGM, `mode: instrumental`) 트랙 전용 비주얼 표준. 자막이 없으므로 정적 커버 위에 **음표 선화 모티프 + 모션 레이어**로 생동감을 준다. **radio(새벽 라디오)가 v1 기준 트랙.**

### 7.1 정적 커버 합성 (음표 선화 모티프)
- 라디오 상단에서 흘러나오는 **선화(line-art) 음표 2~3개** 합성.
- **윤곽선만**(채움 없음), 색 **흰~하늘빛**, **opacity 0.30~0.45**.
- **위로 갈수록 축소**(상승·원근감).
- **제목 영역(하늘 여백) 회피** — 타이포와 겹치지 않게.

### 7.2 모션 레이어 (재렌더 시)
- 동일 음표가 **라디오 위치에서 저속 상승 → 페이드아웃**.
- **동시 존재 최대 3개.**
- 기존 **빛가루 드리프트(particle drift)와 공존.**

### 7.3 타이포·시그니처 (공통 표준 유지)
- 제목 **"새벽 라디오"** 하늘 여백 배치.
- **Reina 사인 우상단** 표준 유지(§3 불변 시그니처).

### 7.4 재승인 규칙
- 모션 빌드로 커버가 바뀌면 **게이트1 재승인 필수**(no stale `.cover_ok` · 재렌더 규칙). 기존 `.cover_ok` 무효화 후 새 커버로 재승인.

### 7.5 현재 상태
- **미빌드(백로그 PLAN §P4).** radio 는 현재 단일 커버 Ken Burns 로만 인코딩되며 `publish_hold: true`(VERIFIED 정지, QUEUED 미투입). 모션 표준 빌드→재렌더 후 발행.
