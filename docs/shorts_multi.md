# 설계안 — make_shorts.py v2 `--segment` 멀티 구간 확장

> 상태: **설계 문서 (코드 미구현)**. Navigator 검수·승인 후 착수.
> 작성 2026-07-13. 작업4(설계) + 작업8(확정 기준) 통합.

## 1. 목적
한 곡에서 여러 훅 구간을 각각 독립 숏츠로 뽑아, 트랙당 숏츠 생산량을 늘리고
같은 곡을 시간차로 재노출한다(임프레션 풀 분산 + 채널 회전율 상승).

## 2. 구간 모델
- 곡당 **최대 3개 구간**: `hook` / `verse` / `bridge`.
  - `hook` = 기존 코러스 자동감지(detect_chorus / repetition / RMS) 결과.
  - `verse` = 첫 벌스 섹션(가사 [Verse] 태그 또는 align.json 첫 블록).
  - `bridge` = [Bridge] 섹션(있을 때만; 없으면 스킵 → 그 곡은 2구간).
- 경계는 전부 **align.json 타임코드 기반**(섹션↔줄 매핑, 기존 parse_sections 재사용).
- 각 구간은 기존 20~40초 창 규칙(최소 길이 우선·억지 연장 금지) 그대로 적용.

## 3. 구간별 필드 (사양 확정 — 작업8)
| 필드 | 값 |
|---|---|
| 오버레이 문구 | **구간별 핵심 가사 1줄**(그 구간에서 가장 대표적인 훅 라인) |
| 제목 형식 | `"가사 구절" \| 곡명 - Reina` (예: `"끝도 없이 달려 동해로" \| 동해로 - Reina`) |
| 하단 핸들 | `@reinamusic_0217` (v2 표준 유지) |
| 파일명 | `<slug>_shorts_<segment>.mp4` (예: `donghae_shorts_hook.mp4`) |

## 4. CLI 설계
```
python make_shorts.py --segment hook   ...   # 단일 구간
python make_shorts.py --segments hook,verse,bridge ...  # 다중(각각 렌더)
--overlay-text "…"     # 구간별 오버레이 가사 1줄 명시(미지정 시 해당 구간 첫 훅줄 자동)
```
- `--segment` 없으면 기존 동작(hook 단일) = 하위호환.
- verse/bridge 감지 실패 시 해당 구간만 스킵 + 경고(전체 실패 아님).

## 5. 발행·로테이션 규칙 (작업8 확정)
- **같은 곡 구간 간 5~7일 로테이션** — 동일 곡의 hook·verse가 최소 5일 간격.
- **신규 곡 기본 2개 구간(hook+verse)**, 3개째(bridge)는 **실적 기반**(1·2번 구간의
  조회수/유지율이 기준 이상일 때만 추가).
- **ledger에 `segment` 필드 추가**: `{track, segment, video_id, ...}` — 구간별 멱등·중복 차단.

## 6. 캐던스 영향 분석 (작업4)
전제: 정상 캐던스 **1일 1건**(2/day 예외는 백로그 소진 한정, 종료됨).

| 시나리오 | 곡수 | 구간/곡 | 총 숏츠 | 1일1건 소진일수 |
|---|---|---|---|---|
| 현재 발행 vocal | 6 (amumaldo·geureoke·songdo·bomnal·donghae·geunal) | 2 | 12 | **12일** |
| + radio(instr, hook만) | 7 | (radio=1) | 13 | 13일 |
| 3개째까지 전량 | 6 vocal | 3 | 18 | 18일 |

**로테이션 제약 반영:** 같은 곡 구간 5~7일 간격이면, 6곡×2구간을 라운드로빈 배치 시
1라운드(각 곡 hook) 6일 → 2라운드(각 곡 verse) 6일, 곡 내 간격 자연히 6일 확보 →
**제약 충돌 없이 12일 큐**가 성립. (3구간이면 18일, 곡 내 간격 6일 유지.)

**결론:** 1일1건 유지 시 현행 6 vocal 트랙만으로 **12~18일치 큐** 확보. 신규 곡이
주기적으로 추가되면 큐가 마르지 않음. 2/day 예외는 재개하지 않아도 파이프라인 지속 가능.

## 7. 구현 범위(승인 후) — 코드 미수정
- `make_shorts.py`: 구간 감지 함수(detect_verse/detect_bridge) 추가, `--segment(s)`·
  `--overlay-text` 인자, 파일명 규칙, 제목 형식 빌더.
- 발행 스크립트/ledger: `segment` 필드, 구간별 멱등, 5~7일 로테이션 스케줄러.
- 회귀 위험: 기존 단일 hook 경로는 기본값으로 보존(하위호환 필수).

## 8. 07-13 확정 표준 수정

### 8.1 해시태그 표준 (작업3)
- 숏츠 해시태그 **3~5개**, 그중 **영어 카테고리 태그 1~2개 필수 포함**.
- 기본 풀(곡 장르에 맞게 선택): `#koreanmusic` `#koreanballad` `#kindie` `#kpopballad`.
- **`#viral` `#fyp` 금지 유지**(스팸 신호). 곡 고유 태그(#곡명 등)와 병행.

### 8.2 컷 길이 · 루프 (작업4)
- **컷 길이 0~32초 → 30~40초** — 자연스러운 가사 단락 경계 기준(억지 컷 금지).
  (현행 make_shorts 기본 20~40 → 멀티구간 스펙은 30~40로 상향; 구현 시 기본값 반영.)
- **루프 설계:** 컷 종료 지점이 **첫 소절로 자연 회귀**하는 구간을 우선 선택
  (무한 루프 시 이음새 자연스러움 = 리텐션 직결).

### 8.3 발행 시각 지터 (작업4)
- `upload_scheduler`에 **발행 시각 ±2~3시간 랜덤 오프셋** 적용(고정 시각 패턴 회피).
  기준 시각 10:00 → 실제 07:00~13:00 사이 랜덤. (구현 시 seed=발행일 기반 결정적 지터.)

### 8.4 3컷째 승격 기준 (작업4, §5 갱신)
- 신규 곡 기본 2컷(hook+verse). **3컷째(bridge) 승격 조건:**
  **Viewed vs Swiped Away ≥ 75%** **AND** **평균 시청률 ≥ 65%** (1·2컷 실측 기준).
